# Core Rules

## Engineering Workflow

- 先讀相關檔案，再修改
- 先做最小修復，再考慮重構
- 修完要列出風險點
- 核心流程：issue建立，確認沒問題再實作，實作確認正確再更新必要內容回AGENTS.MD與README.md

## 開始前先讀

AI 在接手任何任務前，必須先閱讀 [README.md](./README.md) 了解：

- 專案起源是舊系統轉置驗證需求
- 專案用途
- 本輪工作的成功標準
- 原始素材與協作原則

若未先閱讀 `README.md`，不得直接假設專案目標為完整產品化或直接進入過度設計。

## AI 任務入口

接手本專案時，請依序進行：

1. 先讀 [README.md](./README.md) 建立專案背景。
2. 再讀目前要執行的 issue 計畫檔，格式為 `issue/issue-{no}/plan.md`。
3. 如該 issue 已有執行紀錄，續讀 `issue/issue-{no}/impl.md`，避免重複試錯。
4. 若使用者未指定 issue，優先採用最新編號的 issue。
5. 先確認本輪任務屬於：
   - 可行性驗證
   - 不可行性排除
   - 風險釐清
   - 證據整理
   - 正式流程維運與修正
6. 執行時優先追求：
   - 產出明確結論
   - 紀錄證據
   - 幫下一輪減少試錯

## Issue 規則

- 每個 issue 使用 `issue/issue-{no}/` 目錄。
- 計畫一律寫在 `issue/issue-{no}/plan.md`。
- 執行內容、實驗紀錄、結果與 FEEDBACK 一律寫在 `issue/issue-{no}/impl.md`。
- 下一個 issue 依序使用下一個編號，例如 `issue/issue-2/`。
- 若舊路徑仍存在，只作為歷史參考，不作為新的唯一入口。

## AI 工作原則

- 所有分析都要回到「如何支撐新系統測試與驗證」這個主目標。
- 本階段已進入正式使用，但仍以支撐驗證為主，不預設直接擴成過度工程化的大系統。
- 優先回答「是否能穩定支撐實際驗證」，再決定是否值得進一步工程化。
- 對任何自動化判斷，都要保留限制說明。
- 若路線不可行，也算有效成果，但必須寫清楚原因。
- 若路線部分可行，需寫出哪些部分可自動、哪些部分需人工補強。

## Plan 執行規範

所有 `plan` 內的任務，完成時都必須補上 FEEDBACK，不能只寫結果。

每個任務至少要留下以下四個欄位：

```md
### FEEDBACK

- Positive: 這次做法中，哪些做法有效，應該保留。
- Negative: 哪些地方效率差、資訊不足、判斷不穩，或不應再照原方式做。
- Evidence: 這次結論依據了哪些檔案、指標、觀察或樣本。
- Next Run: 下次若再做一次，建議先調整什麼。
```

規則補充：

- `Positive` 不是稱讚，而是要記錄可複用的方法。
- `Negative` 不是責備，而是要記錄踩雷、限制與不划算的路線。
- `Evidence` 要能追溯，不可只寫模糊感受。
- `Next Run` 要可執行，避免只寫抽象建議。

## 現階段輸出期待

本階段任務輸出，優先接受以下類型：

- 可行路線摘要
- 不可行路線摘要
- 風險清單
- 對齊規則草案
- 樣本分析結果
- 可轉為新系統測試或驗證資料的整理結果
- 下一輪建議

本階段不要求：

- 一次完成最終 Web UI
- 一次建完最終資料模型
- 過早拆成過多正式模組

## 決策優先順序

若執行時出現取捨，請用以下優先順序決策：

1. 是否能支撐實際驗證作業
2. 是否有足夠證據支撐結論
3. 是否能讓下一輪更有效率
4. 是否值得再投入工程化

## 文件同步要求

若 AI 更新了計畫、流程或工作規範，應同步檢查：

- `README.md` 的專案階段描述是否仍一致
- `AGENTS.md` 的入口規則是否仍可直接使用
- 目前 issue 的 `plan.md` 與 `impl.md` 是否仍符合 FEEDBACK 要求
