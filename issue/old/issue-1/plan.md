# Issue 1 - 系統翻新比對資料 POC 計畫

本文件由原 `plan/plan-1.md` 遷移而來，作為 `issue-1` 的正式計畫檔。

執行結果請寫入同目錄的 [impl.md](./impl.md)。

## 1. 計畫定位

`issue-1` 是本專案第一輪 `POC` 計畫。

這個 POC 的背景，是舊系統轉置需要一套明確的驗證方法。目標不是只把舊資料拆開，而是要驗證能否透過 `錄影 + HAR + Recording` 三類素材，建立一套足以支撐新系統測試與驗證的資料基底。

本輪目標不是直接完成最終解構平台，而是先找出：

- 哪些路線可行
- 哪些路線不可行
- 哪些路線可行但需要人工補強
- 下一輪應該集中投資在哪裡

因此，本計畫重點在於「驗證路線」而不是「全面交付」。

## 2. POC 核心問題

本輪要優先回答以下問題：

1. HAR 與 Recording 能否可靠對齊到交易層級或主要步驟層級。
2. MP4 是否能切出可用於人工檢核與對照的關鍵畫面。
3. 哪些欄位、節點與流程適合自動抽取，哪些不適合。
4. 是否能做出足以支撐新系統測試與驗證的最小資料結構。
5. 哪些做法成本過高、訊號太弱，應在下一輪避免。

## 3. 本輪成功定義

本輪 POC 若要算成功，至少需滿足：

- 有明確列出的 `可行路線`
- 有明確列出的 `不可行或暫不建議路線`
- 每個任務都有 `Positive` 與 `Negative` FEEDBACK
- 結論能回指原始資料或實驗證據
- 至少能指出哪些輸出可直接轉成新系統測試或驗證材料
- 下一輪的人或 AI 能直接承接，不必重新摸索目標

## 4. 本輪不追求事項

本輪先不以以下項目為主要目標：

- 完整 Web UI 產品化
- 一次建完所有資料模型
- 完整自動化影片辨識
- 一次支援所有情境與所有交易群組
- 過早優化架構、命名與模組切分

如果有局部驗證需要做最小實作，可以做，但要以支撐 POC 判斷為限。

目前已確認來源素材如下：

## 6. POC 輸出格式

每個任務完成後，至少要輸出以下內容：

```md
## 任務名稱

- Goal:
- Method:
- Evidence:
- Decision: feasible / infeasible / partial

### FEEDBACK
- Positive:
- Negative:
- Evidence:
- Next Run:
```

補充規則：

- `feasible` 表示這條路線目前值得繼續投入。
- `infeasible` 表示這條路線目前不值得照原方式推進。
- `partial` 表示局部可行，但需要人工補強、限制條件或額外前置。
- `Positive` 要記錄可複製的方法與成功條件。
- `Negative` 要記錄踩雷、限制、資料缺口與低效做法。
- `Evidence` 要能回指樣本、檔案、指標或觀察依據。
- `Next Run` 要寫成可執行的下一步。

## 7. 工作分流

本輪工作分成五條 POC 路線，每條都要明確產出可行或不可行結論。

### Route A. Recording 結構可用性驗證

目標：確認 Recording 是否足以建立流程骨架。

任務：

1. 盤點 `steps` 結構與主要 `type` 分布。
2. 驗證是否能抽出穩定的流程節點，例如登入、查詢、輸入、確認、送出。
3. 確認 selector、frame、value、target 是否能整理成一致格式。
4. 判斷是否存在足夠時間資訊；若沒有，要明確標示限制。

預期判斷：

- 可行：可以整理出流程骨架，即使沒有精準時間戳。
- 不可行：欄位過度不穩或無法支持主要操作脈絡。

完成時必填 FEEDBACK：

- Positive: 哪些欄位最有助於建立流程骨架。
- Negative: 哪些欄位缺失、格式不穩或語意不足。
- Evidence: 這次判斷依據了哪些 step 樣本或欄位分布。
- Next Run: 下次應先做哪些正規化或抽象化。

### Route B. HAR 重點事件抽取驗證

目標：確認 HAR 是否能抽出對流程與畫面有意義的關鍵事件。

任務：

1. 區分靜態資源與功能事件。
2. 聚焦 document、xhr、fetch、POST、轉導與錯誤頁。
3. 驗證是否能建立功能路徑、狀態碼與異常流摘要。
4. 評估 HAR 是否足以作為流程對照的主要後端證據。

預期判斷：

- 可行：能從大量 entries 中穩定萃取關鍵事件。
- 不可行：噪音過高或無法辨識出主要功能脈絡。

完成時必填 FEEDBACK：

- Positive: 哪類事件最容易提供業務意義。
- Negative: 哪些過濾方式仍保留太多噪音，或哪些欄位無助於判斷。
- Evidence: 這次結論依據了哪些 URL、狀態碼、類型分布或樣本。
- Next Run: 下次應先做哪些白名單、黑名單或聚焦策略。

### Route C. Recording 與 HAR 對齊驗證

目標：確認前端操作與後端事件能否形成可追溯鏈。

任務：

1. 以檔名時間、HAR 起始時間與流程順序建立初步 session 對齊。
2. 使用 `navigate`、頁面切換、提交動作與 URL 特徵做節點對照。
3. 驗證至少一條主要流程是否能從操作步驟追到網路事件。
4. 將對齊結果區分為高信心、低信心、無法對齊。

預期判斷：

- 可行：至少能做到交易層級或主要步驟層級對齊。
- 部分可行：只能做粗粒度對齊，需要人工校正。
- 不可行：缺乏足夠錨點，對齊成本遠高於價值。

完成時必填 FEEDBACK：

- Positive: 哪些錨點最有效，例如檔名時間、navigate、URL pattern、submit action。
- Negative: 哪些情況最容易失配或產生誤判。
- Evidence: 這次對齊是依哪些時間錨點、頁面節點或 request 樣本判定。
- Next Run: 下次應先建立哪些對齊規則或人工確認點。

### Route D. MP4 截圖與檢核價值驗證

目標：確認影片是否值得納入正式流程，還是只適合人工補強。

任務：

1. 驗證能否按時間點穩定切出關鍵畫面。
2. 檢查畫面是否足夠清楚支撐欄位、按鈕、提示訊息或結果頁判讀。
3. 比較固定頻率切圖、關鍵時間點切圖與場景變化切圖的實用性。
4. 判斷影片在 POC 與後續正式階段應扮演的角色。

預期判斷：

- 可行：可穩定擷取並支撐人工檢核。
- 部分可行：只能做關鍵節點輔助，不適合全面自動化。
- 不可行：切圖成本高但資訊增益有限。

完成時必填 FEEDBACK：

- Positive: 哪種切圖策略最有價值。
- Negative: 哪種切圖方式重複高、噪音多或無助於對照。
- Evidence: 這次比較依據了哪些時間點、畫面樣本或切圖結果。
- Next Run: 下次應保留哪些畫面節點，刪除哪些低價值嘗試。

### Route E. 最小檢視輸出驗證

目標：確認是否能先產出最小可讀成果，而不急著做完整產品。

任務：

1. 定義最小輸出，例如流程摘要、關鍵請求摘要、對齊表或 checkpoint 清單。
2. 驗證這些輸出是否足以支撐翻新比對討論。
3. 評估是否真的需要立刻做 Web UI，或 Markdown / JSON 就足夠。
4. 列出進入下一階段前應保留的資料模型骨架。

預期判斷：

- 可行：最小輸出已足以支持討論與下一輪決策。
- 不可行：沒有檢視介面就無法有效使用成果。

完成時必填 FEEDBACK：

- Positive: 哪種輸出格式最容易被人與 AI 共同使用。
- Negative: 哪些輸出太重、太早、太難維護。
- Evidence: 哪些閱讀、比對或討論情境證明這個輸出有效或無效。
- Next Run: 下次應先收斂哪些欄位與視圖。

## 8. 建議執行順序

建議以以下順序進行：

1. 先做 Route A，確認 Recording 可否形成流程骨架。
2. 再做 Route B，確認 HAR 可否形成事件骨架。
3. 接著做 Route C，驗證兩者能否對齊。
4. 視需要再做 Route D，用影片補足與驗證。
5. 最後做 Route E，確認最小成果該長什麼樣子。

這個順序的目的，是先驗證資料層，再決定是否值得做呈現層。

## 10. 驗收方式

本輪驗收不以功能數量為主，而以下列問題是否被回答為主：

1. 哪些路線已證實可行。
2. 哪些路線目前不建議投入。
3. 哪些結果只在特定前提下成立。
4. 下一輪若要做正式化，第一優先該投入哪一段。
5. 每個任務是否都有留下可操作的正負向 FEEDBACK。

## 11. 風險與判讀原則

### 11.1 可能風險

- Recording 缺乏逐步絕對時間，可能只能粗對齊。
- HAR 靜態資源太多，若過濾策略不對會淹沒重點。
- 影片可讀性受畫面清晰度與切圖策略影響。
- 功能代碼、URL 與畫面語意可能需要人工補註。
- 若太早做 UI，可能掩蓋資料層其實尚未穩定的問題。

### 11.2 判讀原則

- 有證據但不完美，優先標記為 `partial`，不要假裝完全可行。
- 沒有足夠證據時，不要硬下定論。
- 若某條路線成本很高但收益有限，可直接列為暫不建議。
- 若結論依賴人工判讀，要明確說明不能當成全自動能力。

## 12. 結論

`issue-1` 的角色是 POC 探路計畫。

本輪的價值不在於做出最大量成果，而在於把可行與不可行的路線清楚分開，留下有證據的正負向 FEEDBACK，讓下一輪不再重複試錯，能更快進入真正值得工程化的方向。
